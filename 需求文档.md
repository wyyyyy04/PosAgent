# POS Template Mapping Agent

## 项目背景

餐饮、奶茶行业存在大量不同的 POS 系统和第三方平台导入模板。

不同模板之间：

* 字段名称不同
* 字段结构不同
* 中英文混用
* 存在冗余字段
* 部分字段为组合字段

目前需要人工完成：

1. 理解模板字段含义
2. 从主数据表查找对应商品
3. 手动映射 SOP
4. 填充配料字段
5. 检查匹配结果

该过程重复、耗时且容易出错。

目标是构建一个：

> POS Schema Mapping Agent

实现：

```text
上传主数据表
上传POS模板
输入一句自然语言指令

↓

Agent自动完成字段理解
Agent自动完成商品匹配
Agent自动填充SOP
Agent生成可导入模板
```

---

# 核心业务目标

目标：

将主数据表中的 SOP 字段自动映射到模板中的配料字段。

最终实现：

```text
CLI:

map sop to 配料字段

↓

自动完成整个工作流
```

# 核心难点

## 1. 字段名称不统一

例如：

```text
商品名称
菜品名称
Product Name
MENU_NM
```

语义一致。

需要 Agent 自动识别。

---

## 2. 字段结构不统一

### 情况1

模板：

```text
温度
糖度
奶底
```

分别独立字段。

---

### 情况2

模板：

```text
口味做法组合
```

字段值：

```text
燕麦奶/正常冰/三分糖
```

多个维度混合在一个字段。

---

## 3. 存在冗余字段

例如：

```text
英文分类
更新时间
备注
分类描述
```

不参与匹配。

Agent需要自动识别并忽略。

---

## 4. 商品名称高度相似

例如：

```text
黑糖波波
黑糖波波牛乳
黑糖波波牛乳茶
```

向量相似度高。

不能仅依赖Embedding。

---

## 5. 奶底可能缺失

主数据：

```text
奶底为空
```

模板：

```text
牛奶
燕麦奶
```

业务规则：

```text
奶底为空
=
通配符
```

允许匹配。

# Canonical Schema

统一内部标准结构：

```json
{
  "product_name": "",
  "size": "",
  "milk_base": "",
  "temperature": "",
  "sugar": "",
  "tea_base": ""
}
```

所有模板最终都转换成该结构。

---

# Schema Analyzer

职责：

理解模板字段。

输入：

```text
字段名
字段样例
```

例如：

字段：

```text
口味做法组合
```

样例：

```text
燕麦奶/正常冰/三分糖
```

输出：

```json
{
  "semantic_type": "composite_field",
  "contains": [
    "milk_base",
    "temperature",
    "sugar"
  ],
  "importance": "high"
}
```

---

# Token Parsing

目标：

把组合字段解析成标准结构。

例如：

```text
大杯牛奶正常冰三分糖绿茶
```

解析：

```json
{
  "size": "大杯",
  "milk_base": "牛奶",
  "temperature": "正常冰",
  "sugar": "三分糖",
  "tea_base": "绿茶"
}
```

# 商品匹配策略

## 商品名称单独匹配

商品名称不参与组合字段解析。

原因：

```text
黑糖波波
黑糖波波牛乳
黑糖波波牛乳茶
```

存在实体歧义。

---

## 属性单独匹配

匹配：

```text
规格
奶底
温度
糖度
茶底
```

---

## 最终定位

```text
商品名
+
属性组合

↓

唯一定位SOP
```
